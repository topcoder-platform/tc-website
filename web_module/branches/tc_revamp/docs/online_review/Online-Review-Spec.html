Online Review Requirements Specification

1.     Scope

The Online Review Application will take automates the process of the TopCoder Software’s peer review process process for both Component Design and Component Development projects.   Reviewers will beare able to login, check the status of their assignments, download submissions for review, fill-incomplete online review forms and perform aggregation of reviews.  There is an The administration section that allows users to set up reviews, assign members, monitor the process and intervene at any stage and score components of development.

 

 

1.1     Logic Requirements

1.1.1     Reviewers Track Assignments

Reviewers can login and view current board assigned projects (components), including dates of each portion of the review and their role in the process.  From this section, reviewers must be able to navigate easily to their current action items.

1.1.2     Submission Upload

All developers/designers must be able to upload submissions for any projects they have inquired about.  The system must track each user’s submissions and assign each submitter a unique id for each project to which they submit solutions (i.e. user XXX submitting for projects Y and Z might be assigned “Submitter 1500” and Submitter 1501” respectively, but if user XXX submits a correction for project Y, they are still “Submitter 1500” and the file must be stored accordingly).  These unique ids must be unique system-wide. Submitters must be referred to by this generic name throughout the application to maintain their anonymity during review.

1.1.3     Primary Reviewer Design and Development Screening Scorecard

Primary Reviewers will have an on-line tool to fill out the screening scorecard for each submission that provides access to all required files and a checklist of screening requirements and a place to enter text for responses in response to others.  The text of each of these questions must be configurable (see Configuration).

 

Once screening is completed, the Product Manager must be able to view and edit the results, select the winners based upon the scores and send email notifications to the reviewers.

1.1.4     Development Reviewers and Test Case Approval – what does this have to do with Design Reviewers?

All reviewers will have an online tool to approve or reject the test cases submitted by fellow review board members.  This requires yes/no approval and a place to enter text responses (to be sent to the Product Manager) if warranted.

1.1.5     Reviewers Design and Development Scorecard

All reviewers will have an on-line tool to fill out either the design or development scorecard whichthat provides access to all required submissions.  Scorecards must be capable of presenting multiple sections consisting of:

1.       Description of attribute to be scored (herein referred to as Questions).

2.       One overall rating response to each question.

3.       Ability for reviewer to provide zero or many responses to each question.

4.       Require at least one response for any question eliciting a less-than-perfect overall rating.

5.       Responses must consist of:

a.       Text response.

b.       Decision on whether response describes a requirement, recommendation or other comment.

Scorecards must be able to be saved for later completion. Individual scorecards may be marked as complete (although this does not preclude further editing).  Reviewers must be able to submit all scorecards once each individual review is complete, indicating the reviews are ready to be used in the scoring phase.  If reviewers have marked all scorecards as complete, but have not submitted, when the end of the review process is reached, the scorecards are submitted.

 

1.1.6     Primary Reviewer Aggregation

When all reviews are received, the winner is calculated using a configurable weighted matrix (see Configuration) and the Primary Reviewers have an online tool to aggregate all review commentsresponses.  This tool will showpresent the winning submission for the all component, each question from the review and all reviewer responsescomments and give t. The primary reviewer must have the ability to mark each individual response as “Accepted”, “Rejected” or “Duplicate” as well as the ability to accept, reject, edit the reviewers commentsresponses or switch it’s status between required/recommended/comment.

 

When completed, reviewers must be notified and provided with a means of accepting/rejecting the aggregation and providing text comments upon rejection.

 

Once accepted, the Product Manager must be able to view and edit the results and send the notification emails. 

Add the ability to reject the winner. This would be in case a winner is in-eligible, or does not complete the final review.

1.1.7     Final Review

Primary Reviewer must have an online tool to review the ‘final’ submission of the winning developer.  This tool must provide access to all required files as well as presenting the reviewer with the questions and responses recorded for the developer (see 1.1.5) along with a method for indicating whether each has or has not been completed in the revision.

Admins must have aAdmin tool to set upspecify begin and end dates for each phase of the review process.  review milestones and perform board assignments.  These dates serve as a guide to govern the scheduling of above-described phases.

 

  The reason should be a drop down box with a choice, as well as a comment section (thoughts?).Admins must be able to allocate resources to each phase of the project by entering the handle of the developer to perform the specified task.  Primary reviewer must be indicated.  Emails notifying reviewers of initial assignments must be sent once a week after all assignments are completed (to minimize the number of emails and provide consistency).  Notification emails based upon configurable templates (see Configuration) must be sent upon modification.

1.2     Design Specification

1.2.1     Design Overview

This design refers to all artifacts of the review process (submissions, scorecards etc) as ‘documents’.

 

The creation of projects is outside the scope of this project and, as the point of project creation is the most logical time to create the roles for a given project, this appears to fall outside the scope of this project as well.  The same applies to the installation of inquiring members as submitters for a project as this ought to occur when a member inquires, which is also outside the defined scope of this project.  However, direction is provided within the persistence layer implementation to ease the implementation of such functionality.

 

The HTML User Interface for this application will consist of JSP pages tied together by the MVC Framework Component.  The Controller checks the intended destination or desired action and passes the request to the appropriate Model implementation.  The Model checks for proper authorization (to be discussed later), performs all validation and then processes the request as described in the sequence diagrams.

The Security Manager will be used to authenticate users as well as to authorize their activities within the system.  The login process is entered through Login.jsp which hands off control to LoginModel, which makes an authentication request to the AuthenticationBean.  If the AuthenticationBean fails to authenticate the user, they are redirected back to Login.jsp.  Any non-login requests cause the system to check for existence of a session containing a valid user principal.  If such a principal is not found, the user will be redirected to Login.jsp where they are prompted for a username and password.  If a user principal is found, it is checked for authorization to view/use the requested page.  If the user selects “Logout,” the user principal is removed from their session and they are redirected to the login page.

Each page is itself rendered using the user’s roles as a reference for which controls are visible and which are not (users will not be presented with non-functional or greyed-out controls).  All information which is not related to security (or user principal) will be passed from page to page in the request object to allow users to work in multiple windows at one time without adverse effects.  The individual pages contain the logic to represent the proper views to each user as the page has access to the current phase and user (they are stored in the session at request level by the model).  The Object Formatter component provides standardized representation of all data objects across all pages.

 

This Application consists of two stateless session beans, the ProjectTrackerBean and the DocumentManagerBean.  These beans provide container-managed transactions.  These beans will execute custom SQL to retrieve and store the data used using the managed value objects.  All SQL used in this application is in the form of prepared statements executed inside these beans in an effort to facilitate software maintenance by locating all database access in easily identifiable (Manager) objects and minimize overhead while still separating the business logic from the persistence layer.  The appropriate manager handles the retrieval and persistence of these value objects.  This convention applies also the implementations of Refreshable Cache.

 

The implementations of Refreshable Cache provide access to read-only data to the system, and therefore reads will be infrequent while value object instance creations or modifications are unsupported through the system (on the other hand, the source data may be changed via external means necessitating the use of a Refreshable Cache).  This read-only data consists of lookup tables stored in the database in an effort to avoid reliance upon constants stored within the Java classes.  Examples of such data include project phase (Phase), payment status (PaymentStatus), and role (Role).  Each of these is accessed through the corresponding manager object.  Evaluations are also managed in this way, but they come in two forms, subjective (‘strongly agree’, ‘agree’, ‘disagreee’ or ‘strongly disagree’) and objective (‘yes’ or ‘no’), and are retrieved using the corresponding methods.

 

The Project Tracker keeps track of the current phase of and provides access to all project metadata.  This includes users, timeline information, payment status and assignments.  This data is used by the “View My Open Projects” Model and View (JSP) to determine the user’s current responsibilities based upon their role and the current phase of the project and present them with navigation options accordingly (i.e. Reviewer for Project X is presented with a link to the Review Scorecards from the Project X detail page when it is in the review phase).  When a user opens a “Project Detail” page, the Project ID is included in each subsequent project page until they access another project’s detail page.  Each Project maintains a URL which is intended to contain a link to more information regarding a particular project (to extend the TopCoder Component Development example, this would contain a link to the component’s forums).

 

The same Model object will handle all activities within a particular use case.  For instance, when the user navigates to “Contact Product Manager,” this request as well as the form submission request is to be handles by the ContactProductManager model.

 

All objects in the persistence layer select the requested objects with the latest ENTRY_DATE unless an ID is specified.  ID’s are included in all retrieved information and will be used in future retrievals for performance.  The persistence layer will handle all data persistence as creation of new rows within the database using the updated information and the current date/time.

 

When files are uploaded by designers, these files are renamed using the format “Submitter_<<[SUBMISSION].[User_ROLE_ID]>>_<<[SUBMISSION].[ENTRY_DATE]>>” and the date will be formatted by a DateFormatMethod statically retrieved from FormatMethodFactory with “yyyy-MM-dd-HH-mm-ss-SS” as the format string.  When the submissions are provided to the screeners/reviewers/aggregators for download, the ENTRY_DATE portion will be stripped off and they will only be provided the latest submission.  The File Upload Component will handle retrieving the file data from the HTTP Request object.  The File Upload component simplifies the retrieval of files uploaded as part of an HTTP Request and is therefore retrieved when populating the MVC ActionData object object, however, it’s location on the file system (including naming) is determined and set within the business logic or Model implementation.  The same process is used in accepting the winning designers’ submission of Final Fixes.  Files are stored on a shared networked drive to enable access to file data from each member of the cluster.

 

The DocumentManager keeps track of and provides access to all scorecards and reviews constructed by either the Product Manager, the Reviewers or the Submitters.  Upon document retrieval, the DocumentManager initializes the retrieved documents with the user requesting the document so it can determine which table to store new data to.  This also enables the document to correctly populate the MODIFIER_ID field.  The Online Review documents require only simple list management in the form of wholesale get/set methods as necessary.  The business logic and user interface logic then use the provided elements directly.  During save operations, the consistency of these data elements is verified for proper quantity and sequence.  If any of these are violated, InvalidEditException (or a subclass thereof) is thrown.

 

The application does not allow modifications to documents by non-admin users once they have been successfully submitted.  This is enforced through the user interface and checked again within the DocumentManager.  The system also does not allow projects to be advanced unless all the required documents have been submitted to ensure that all documents are in stable states.  Admins have the ability to submit any document that has not yet been submitted at any time to avoid the situation where an out-of-touch reviewer might hold up the progress of a review.

Once a document has been submitted, the system (user interface and business logic both) will not present the user with any opportunities to edit the document (previously editable fields will be displayed as read-only HTML).  However, in the event that access to such a submitted document is gained and an attempt is made to save modified data, the system will throw a DocumentAlreadySubmittedException causing the user interface to redirect to an error page indicating the situation.

 

The templates used by the DocumentManager when it generates the required scorecards are versioned at both the question and template level to insure that the template used to construct a scorecard is always accessible.  No interface (aside from direct SQL statements executed against the database) is provided by the online review application to support editing of these scorecard templates.  This versioning is supported by the presence of a modification timestamp as well as the modifying user (the same as is found in the scorecards themselves) within the data structure.  Upon a project’s advancement to the screening phase, the screening scorecard is immediately generated from the existing template in its current state and upon reaching the review phase, the required review scorecards are generated with links to the template version from which they were generated.

 

The Job Scheduling component is used to schedule the delivery of weekly emails to review board members informing them of their assignments.  The purpose in selecting the Job Scheduler for this activity is to allow the compilation of these assignments into one email and avoid flooding users mailboxes with incremental changes.  The scheduler calls the main entry point to ReviewerAssignmentNotification which executes sql statements directly (rather than use ProjectTracker) as this is a bulk data load and instantiation all the related objects would use considerable resources unnecessarily.

1.2.6     Exception Handling Overview

If a user attempts to “Deep Link” or reach a page directly without going through the normal workflow, if logged in, they will be redirected to an error page (Error.JSP not provided) explaining the policy and provided with a link to “My Open Projects.”  If this occurs with a user that is not logged in, the error page will explain the policy, indicate that they are not yet logged in and provide them with a link to the Login.JSP.

Failed authorization attempts will redirect to an error page indicating the situation and providing a link to the login page as well as ViewMyOpenProjects.

Any system-level Exceptions or Errors encountered during execution (SQLException, EJBException, RMIException etc.) will redirect to the error page requesting that they retry the operation later.

The Model-View-Controller framework provides a mechanism for checking for and handling invalid data.  Invalid data provided to this application will redirect the user back to the referring page and highlighted text will indicate the problematic input.

An application-level exception (i.e. IncorrectProjectStateException, DocumentAlreadySubmitted, IncorrectProjectState or any subclasses) will cause the user interface to redirect the user to error.jsp, which will explain what happened.  After such an event, the user will be directed to the “My Projects” page.
